# 设计模式：软件设计原则与 GoF 23 种模式

## 1. 软件设计原则

**一姐弟合理单开**

软件实体（类、模块、方法、接口等）

### 1.1 开闭原则（OCP）

**核心定义**：软件实体应当对扩展开放，对修改关闭。

**核心目的**：在不改动原有稳定代码的前提下扩展新功能，降低引入缺陷风险和维护成本。

### 1.2 单一职责原则（SRP）

**核心定义**：一个软件实体应围绕同一职责组织，并且只应有一个引起其变化的原因。

**核心目的**：降低职责耦合，提升代码可读性、可维护性和问题定位效率。

### 1.3 接口隔离原则（ISP）

**核心定义**：客户端不应当依赖其不需要的接口方法，应使用细粒度、专用化的接口。

**核心目的**：减少无效依赖，降低接口冗余，提升接口灵活性、适配性。

### 1.4 里氏替换原则（LSP）

**核心定义**：子类型对象应能够替换其父类型对象，并且替换后不破坏调用方依赖的行为约定。

**核心目的**：保证继承体系的合理性，规范继承使用，避免子类破坏父类原有行为约定。

### 1.5 依赖倒置原则（DIP）

**核心定义**：高层模块不依赖低层模块，二者都应依赖抽象；抽象不依赖细节，细节依赖抽象。

**核心目的**：弱化模块间耦合度，提升代码扩展性、可替换性，便于后续功能迭代与底层实现替换。

### 1.6 合成复用原则（CRP）

**核心定义**：优先通过组合和聚合实现代码复用，而不是继承。

**核心目的**：降低类与类之间的耦合，避免继承带来的冗余约束，提升代码灵活性与扩展性。组合（整体包含部分）、聚合（整体持有部分）是松耦合复用。

### 1.7 迪米特法则（LOD）

**核心定义**：最少知道原则，一个对象应尽量少了解其他对象，只与直接相关的对象通信。

**核心目的**：降低对象间耦合，简化交互关系，提升系统稳定性和可维护性。

## 2. 设计模式

**抽工建单元，世代想外租装桥，**

**蝶姐观测中，责备模仿命令状。**

### 2.1 创建型模式（5 种）

关注对象如何创建，解耦对象的实例化过程。

#### 2.1.1 单例模式（Singleton）

**核心定义**：保证一个类在运行过程中仅有一个实例，由该类创建和持有该实例，对外提供一个全局访问点。

**核心目的**：避免对象重复创建，减少资源开销；确保全局状态一致；统一访问入口。

**设计原则**：遵循单一职责、迪米特法则。

**创建分类**：饿汉式（直接创建）、懒汉式（使用时再创建，要考虑线程安全）

#### 2.1.2 简单工厂模式（Simple Factory）

不属于 GoF（Gang of Four）官方定义的 23 种设计模式。

**核心定义**：由工厂类封装对象创建逻辑，通过内部条件判断返回不同的具体产品，对外暴露统一抽象产品接口。

**核心目的**：将对象创建与使用分离，降低调用方对具体类的依赖，统一产品创建入口。

**设计原则**：遵循单一职责原则、迪米特法则；违反开闭原则（新增产品问题）。

#### 2.1.3 工厂方法模式（Factory Method）

**核心定义**：将对象创建行为抽象为工厂接口，由具体工厂子类决定具体产品的实例化，对外只暴露统一的创建接口。

**核心目的**：将具体对象创建延迟到子类实现，解耦对象创建与使用过程，使高层模块依赖抽象而非具体实现。

**设计原则**：遵循单一职责原则、开闭原则、依赖倒置原则。

#### 2.1.4 抽象工厂模式（Abstract Factory）

**核心定义**：为一组相关或相互依赖的产品提供统一的创建接口，由具体工厂子类负责创建同一产品族中的多个具体对象。

**核心目的**：将多类关联对象的创建统一管理，保证同一产品族中的对象配套使用，降低客户端对具体实现的依赖。

**设计原则**：遵循单一职责原则、开闭原则、依赖倒置原则。

#### 2.1.5 建造者模式（Builder）

**核心定义**：将复杂对象的构建过程与其具体表示分离，在固定构建流程下，通过不同的具体步骤配置生成不同对象。

**核心目的**：将复杂对象的创建过程拆分管理，提升构建过程的可控性、复用性和扩展性。

**设计原则**：遵循单一职责原则、开闭原则。

**模式对比**：建造者模式用于复杂对象创建流程，模板方法模式用于统一业务流程。

#### 2.1.6 原型模式（Prototype）

**核心定义**：通过复制现有原型对象来创建新对象，而不是通过重新实例化生成对象，复制分为浅拷贝和深拷贝。

**核心目的**：减少复杂对象重复创建成本，提高对象创建效率，并保持新对象与原型对象结构一致。

**设计原则**：遵循单一职责原则、开闭原则。

**拷贝分类**：浅拷贝只复制对象本身及其引用地址，深拷贝会连同引用对象的内容一并复制。

### 2.2 结构型模式（7 种）

关注类 / 对象如何组合成更大结构。

#### 2.2.1 适配器模式（Adapter）

**核心定义**：将不兼容的接口转换为目标接口，使原本不能直接协作的类能够统一接入和使用。

**核心目的**：兼容已有接口差异，复用现有功能，实现上层对不同实现的统一调用。

**设计原则**：遵循单一职责原则、开闭原则、迪米特法则。

#### 2.2.2 桥接模式（Bridge）

**核心定义**：将两个独立变化维度拆分为抽象层和实现层，通过组合关系连接，使两者可以独立扩展。

**核心目的**：解耦抽象与实现，支持不同维度自由组合，避免多层继承导致的类数量膨胀。上层不直接面向具体类编程，而是通过抽象层间接调用底层实现能力。

**设计原则**：遵循开闭原则、单一职责原则、依赖倒置原则。

```c
#include <stdio.h>

/* 实现层：通信接口抽象 */
typedef struct CommOps {
    void (*send)(const char *data);
} CommOps;
/* 具体实现：UART */
void uart_send(const char *data) {
    printf("[UART] send: %s\n", data);
}
/* 具体实现：I2C */
void i2c_send(const char *data) {
    printf("[I2C] send: %s\n", data);
}

/* 抽象层：业务逻辑 */
typedef struct DeviceService {
    CommOps *comm;
} DeviceService;
/* 业务1：设备激活 */
void device_activate(DeviceService *service) {
    service->comm->send("ACTIVATE");
}
/* 业务2：参数下发 */
void device_send_config(DeviceService *service) {
    service->comm->send("SEND_CONFIG");
}

int main(void) {
    CommOps uart_ops = { uart_send };
    CommOps i2c_ops  = { i2c_send };

    DeviceService uart_device = { &uart_ops };
    DeviceService i2c_device  = { &i2c_ops };

    device_activate(&uart_device);
    device_send_config(&i2c_device);

    return 0;
}
```

#### 2.2.3 组合模式（Composite）

**核心定义**：将叶子节点和容器节点统一抽象，组织成整体与部分一致的层次结构，使单个对象与组合对象能够被统一使用。

**核心目的**：简化树形结构处理方式，提升整体与局部操作的一致性，降低客户端对节点类型差异的感知。

**设计原则**：遵循开闭原则、单一职责原则、依赖倒置原则。

#### 2.2.4 装饰器模式（Decorator）

**核心定义**：通过对对象进行嵌套包装，在不修改原有类的基础上，为对象动态叠加新功能。

**核心目的**：在保持原有实现稳定的前提下实现功能扩展，支持按需组合能力，避免继承扩展带来的类膨胀。

**设计原则**：遵循单一职责原则、开闭原则；通过组合代替继承，增强扩展灵活性。

**模式对比**：装饰器模式侧重增强对象功能（加功能），代理模式侧重控制对象访问（加控制）。

#### 2.2.5 外观模式（Facade）

**核心定义**：将子系统中分散且复杂的调用流程统一封装为一个对外接口，向客户端提供简化访问入口。

**核心目的**：降低系统使用复杂度，减少客户端与子系统的耦合，符合最少知道原则。

**设计原则**：遵循迪米特法则、单一职责原则。

#### 2.2.6 享元模式（Flyweight）

**核心定义**：将大量对象中重复、通用且不变的部分抽离为共享享元对象，供多个对象复用。

**核心目的**：减少重复对象创建，降低内存占用，提高系统资源利用率。

**设计原则**：遵循单一职责原则、开闭原则；通过共享机制提升对象复用性。

#### 2.2.7 代理模式（Proxy）

**核心定义**：通过代理对象接管对目标对象的访问，在不改变目标对象的前提下，对外增加一层统一的访问与管理控制。

**核心目的**：在访问目标对象时附加控制能力，如权限校验、延迟创建、缓存、日志、远程访问和生命周期管理。

**设计原则**：遵循开闭原则、单一职责原则、迪米特法则。

**模式对比**：装饰器模式侧重增强对象功能（加功能），代理模式侧重控制对象访问（加控制）。

### 2.3 行为型模式（11 种）

关注对象间如何通信、职责分配。

#### 2.3.1 责任链模式（Chain of Responsibility）

**核心定义**：将多个处理节点串联成一条处理链，请求沿链依次传递，由各节点判断是否处理或继续下发。

**核心目的**：解耦请求发送方与处理方，提升处理流程的灵活性和扩展性，支持动态调整处理顺序。

**设计原则**：遵循单一职责原则、开闭原则、迪米特法则。

#### 2.3.2 命令模式（Command）

**核心定义**：将请求封装为独立的命令对象，使操作请求、参数和执行逻辑解耦，并支持统一调度与管理。

**核心目的**：降低请求发送方与执行方耦合，便于实现命令排队、撤销恢复、日志记录和统一控制。

**设计原则**：遵循单一职责原则、开闭原则、迪米特法则。

**补充理解**：命令模式常配合队列、撤销、重做、日志和任务调度机制使用。

#### 2.3.3 迭代器模式（Iterator）

**核心定义**：将集合对象的遍历行为封装为独立迭代器，在不暴露内部结构的前提下，按统一方式访问元素。

**核心目的**：分离集合存储与遍历逻辑，统一遍历接口，降低客户端对集合内部实现的依赖。

**设计原则**：遵循单一职责原则、开闭原则、迪米特法则。

**补充理解**：将集合的访问路径、游标移动和结束判断等遍历规则封装起来，对外提供统一遍历接口（本质上就是遍历算法类）。

#### 2.3.4 中介者模式（Mediator）

**核心定义**：引入中介对象统一封装多个对象之间的交互逻辑，将对象间的多对多关系转为一对多关系。

**核心目的**：减少对象之间的直接耦合，集中管理复杂交互，提升系统可维护性和扩展性。

**设计原则**：遵循迪米特法则、单一职责原则、开闭原则。

**模式对比**：外观模式是对外简化子系统调用，中介者模式是对内协调多个对象交互。

#### 2.3.5 备忘录模式（Memento）

**核心定义**：在不破坏对象封装性的前提下，保存对象某一时刻的内部状态，并在需要时恢复到该状态。

**核心目的**：支持状态备份、撤销与回滚操作，分离状态存储与业务处理逻辑。

**设计原则**：遵循单一职责原则、迪米特法则。

#### 2.3.6 观察者模式（Observer）

**核心定义**：定义对象间的一对多依赖关系，当主体状态发生变化时，自动通知并更新所有已订阅的观察者。

**核心目的**：解耦主体与观察者，实现状态变更后的事件联动与消息广播。

**设计原则**：遵循开闭原则、单一职责原则、迪米特法则。

#### 2.3.7 状态模式（State）

**核心定义**：将对象在不同状态下的行为封装到独立状态类中，由当前状态负责对应业务处理，并决定状态流转。

**核心目的**：消除大量条件分支判断，解耦状态切换与状态行为，提升状态相关逻辑的可维护性和扩展性。

**设计原则**：遵循单一职责原则、开闭原则、状态封装思想。

#### 2.3.8 策略模式（Strategy）

**核心定义**：将一组可互换的行为逻辑分别封装为独立策略，并根据条件选择具体策略执行同一类业务目标。

**核心目的**：分离不同算法或行为实现，减少条件分支判断，提升逻辑切换的灵活性和扩展性。

**设计原则**：遵循单一职责原则、开闭原则、依赖倒置原则。

**模式对比**：简单工厂模式侧重返回不同对象，策略模式侧重选择不同行为逻辑。

#### 2.3.9 模板方法模式（Template Method）

**核心定义**：在抽象类中定义固定的执行流程框架，将可变细节步骤延迟到子类实现，使流程不变、细节可扩展。

**核心目的**：统一业务执行流程，减少重复代码，规范处理步骤，使子类只关注差异化实现。

**设计原则**：遵循开闭原则、里氏替换原则、单一职责原则。

**补充理解**：在嵌入式开发中，常由上层统一调度固定流程，仅将差异化步骤抽象为函数指针或操作接口，由不同实现按需替换具体细节。

**模式对比**：模板方法模式用于统一业务流程，建造者模式用于组织复杂对象创建流程。

#### 2.3.10 访问者模式（Visitor）

**核心定义**：将数据结构与作用于其上的操作行为分离，通过访问者统一封装对多种元素类型的处理逻辑，新增行为时无需修改元素结构。

**核心目的**：分离数据结构与业务操作，便于在稳定元素结构上扩展新的处理行为，统一管理多类元素的访问逻辑。

**设计原则**：遵循开闭原则、单一职责原则、依赖倒置原则。

**补充理解**：前提是元素类型相对稳定，而操作行为经常变化，功能逻辑由访问者实现。

#### 2.3.11 解释器模式（Interpreter）

**核心定义**：定义一套简化语法规则，并为其建立解释器结构，将特定格式的文本、表达式或指令解析为可执行的程序逻辑。

**核心目的**：将复杂输入规则结构化处理，实现对特定语言或表达式的统一解析与执行。

**设计原则**：遵循单一职责原则、开闭原则。
