# 裸机并发架构笔记：OSAL 与 LTX 的事件调度设计

本文是一篇持续维护的裸机并发架构笔记。当前先分析 OSAL 和 LTX，后续再按同一套维度加入 Protothreads 等方案。重点不在 API 罗列，而在回答几个更基础的问题：谁保存状态、谁产生事件、调度器怎样找到工作、延时如何实现、系统什么时候能够休眠。

分析基于以下本地源码快照：

- OSAL：`mcuwty/osal`，提交 `9fabed3`（2019-09-27）；
- LTX：`TiX233/ltx` V3.3，提交 `5d5b794`（2026-07-07）。

两者都不是抢占式 RTOS。它们没有为每个任务保存独立调用栈，也不会在普通回调执行到一半时切换到另一个回调。更准确地说，它们都是以**运行到完成**为基本约束的协作式事件框架。

---

## 1. 先建立统一的分析视角

### 1.1 裸机真正需要组织的内容

简单轮询可以把所有工作写进一个 `while (1)`：

```c
for (;;) {
    scan_key();
    update_sensor();
    run_protocol();
    refresh_display();
}
```

问题不只是循环越来越长，而是时间关系、模块依赖和状态推进混在了一起：

- 快任务会被慢函数拖延；
- 每个模块都要自行判断“是否到了执行时间”；
- 中断与主循环之间散落着大量标志位；
- 很难判断系统此刻是否真的空闲；
- 一个业务流程跨越多个时间点时，只能手写状态机或者阻塞等待。

OSAL、LTX 和以后要讨论的 Protothreads，本质上都在尝试把这些问题结构化，但它们选择的核心对象不同。

### 1.2 五个核心问题

分析一种裸机架构时，可以先问下面五个问题：

| 问题 | 观察内容 |
|------|----------|
| 谁保存状态 | 任务结构、事件位、话题对象，还是函数内部的续执行位置 |
| 谁产生事件 | 中断、定时器、其他任务或外设回调 |
| 如何找到工作 | 遍历任务、弹出就绪队列，还是恢复某个协程 |
| 如何表达等待 | 返回主循环、登记闹钟、订阅事件或保存行号 |
| 如何进入空闲 | 忙轮询、就绪判空、Idle 任务或 Tickless |

这组问题比“有几个 API”更能揭示一个框架的成本和适用边界。

## 2. 共同基础：事件驱动与运行到完成

### 2.1 中断只生产事件，前台消费事件

两种框架都适合采用下面的分工：

```text
硬件中断 / DMA 回调 / SysTick
            │
            │ 置事件、发布话题、投递消息
            ▼
       待处理状态或就绪队列
            │
            │ 调度
            ▼
       前台事件处理函数
```

中断负责确认硬件状态、搬走必要数据并通知前台，复杂业务留给前台执行。这样既能缩短中断占用时间，也能把大部分状态变化限制在可控的调度上下文中。

### 2.2 回调必须主动交还执行权

两者的调度器都可以抽象成下面的伪代码：

```c
for (;;) {
    work = get_next_ready_work();
    if (work != NULL) {
        run_to_completion(work);
    } else {
        idle_or_poll();
    }
}
```

`run_to_completion()` 是最重要的约束：一个处理函数不返回，其他同级工作就没有机会运行。因此，回调中不能使用长延时、等待外设状态的死循环或不可控的大计算。需要等待时，应保存状态、登记下一次事件，然后立即返回。

这类框架提供的是**并发组织方式**，不是抢占。中断仍然可以抢占前台代码，但两个普通任务或订阅者之间不会发生 RTOS 式的上下文切换。

## 3. OSAL：以任务和事件位为中心

### 3.1 数据模型

OSAL 为每个任务保存初始化函数、事件处理函数、任务 ID、优先级和 16 位事件集合。源码中的核心结构可简化为：

```c
typedef struct OSALTaskREC {
    struct OSALTaskREC *next;
    void   (*init)(uint8_t task_id);
    uint16_t (*process)(uint8_t task_id, uint16_t events);
    uint8_t  task_id;
    uint8_t  priority;
    uint16_t events;
} osal_task_t;
```

任务按优先级从高到低插入单链表；当前实现中，优先级数值越大，位置越靠前。这里的优先级只决定调度器**下一次选择谁**，不会抢占正在运行的低优先级任务。

### 3.2 调度路径

主循环每次从任务链表头开始，找到第一个事件位非零的任务。取走事件后调用处理函数，再把函数返回的未处理事件合并回任务：

```c
for (;;) {
    task = osalNextActiveTask();
    if (task == NULL) {
        continue;
    }

    enter_critical();
    events = task->events;
    task->events = 0;
    exit_critical();

    remaining = task->process(task->task_id, events);

    enter_critical();
    task->events |= remaining;
    exit_critical();
}
```

任务链表已经按优先级排序，所以第一个就绪任务就是当前最高优先级任务。但查找仍需从链表头开始：最坏情况下每次调度要检查全部 `T` 个任务，空闲时也会不断进行这次扫描。

源码注释提到无事件时可以休眠，但当前 `osal_start_system()` 没有实际的 Idle Hook 或休眠调用，因此默认行为是忙轮询。移植到低功耗设备时，需要补上安全的判空和休眠路径。

### 3.3 事件位的语义

`osal_set_event()` 将新事件按位或到目标任务：

```c
task->events |= event_flag;
```

一次调度可以同时携带多个事件。处理函数处理某一位后，应返回其余尚未处理的位：

```c
uint16_t sensor_process(uint8_t task_id, uint16_t events)
{
    if (events & EVT_SAMPLE_READY) {
        consume_sample();
        return events & (uint16_t)~EVT_SAMPLE_READY;
    }

    return events;
}
```

这段返回协议不能忽略。调度器在调用前已经把任务的事件字段清零；如果处理函数无条件返回 `0`，本次传入但尚未处理的其他事件就会丢失。

事件位表达的是“至少发生过一次”，而不是发生次数。同一个事件在被处理前连续置位十次，结果仍然只有一个 bit。因此它适合表达状态变化或唤醒条件，不适合直接统计脉冲次数。需要保留每次发生及其数据时，应使用计数器或消息队列。

### 3.4 软件定时器

OSAL 的每个软件定时器保存剩余时间、事件位、目标任务和可选重装值。每个硬件 Tick 都遍历整个定时器链表并递减：

```c
for (timer = timer_head; timer != NULL; timer = next) {
    timer->timeout = saturating_sub(timer->timeout, elapsed);

    if (timer->timeout == 0) {
        osal_set_event(timer->task_id, timer->event_flag);
        reload_or_remove(timer);
    }
}
```

设活跃定时器数量为 `A`，每次 Tick 的工作量为 `O(A)`。优点是实现直观、周期定时器天然支持重装；代价是即使所有定时器离到期还很远，系统仍要被固定 Tick 唤醒并扫描整个链表。

定时器节点来自 OSAL 内存池。创建、停止和到期都会涉及链表与内存管理，因此中断临界区长度、内存池容量和碎片情况都属于移植时需要验证的内容。

### 3.5 消息队列

事件位只能通知“有事发生”，消息队列则保存每一次消息及其负载。OSAL 在用户数据前放置内部消息头：

```text
[ next | len | destination ][ user payload ... ]
                              ^
                              返回给调用方的指针
```

发送流程是：从 OSAL 堆申请消息、填充负载、追加到全局消息队列，再给目标任务置 `SYS_EVENT_MSG`。目标任务收到事件后循环取走属于自己的消息，处理完必须释放：

```c
if (events & SYS_EVENT_MSG) {
    message_t *msg;

    while ((msg = (message_t *)osal_msg_receive(task_id)) != NULL) {
        handle_message(msg);
        osal_msg_deallocate((uint8_t *)msg);
    }

    return events & (uint16_t)~SYS_EVENT_MSG;
}
```

这让 OSAL 同时拥有两种不同语义：

- **事件位**会合并，成本低，用来唤醒任务；
- **消息**逐条排队，可携带快照数据，但需要管理容量、所有权和释放。

当前实现使用一条全局单链表。队尾追加和按目标任务接收都可能扫描消息链表，复杂度为 `O(M)`；消息很多时，队列操作和临界区会随之变长。

### 3.6 OSAL 的设计取舍

OSAL 的优势是任务边界明确，事件位、消息、软件定时器和内存池组成了一套完整的小型运行环境。业务很容易按“某任务收到某事件”来组织，优先级规则也容易理解。

它的主要成本同样来自“以任务为中心”：

- 设置事件前要按 ID 查找任务，最坏为 `O(T)`；
- 每次选择就绪任务都要扫描任务链表，最坏为 `O(T)`；
- 每次 Tick 都扫描活跃定时器，成本为 `O(A)`；
- 消息路径依赖动态内存和全局链表；
- 16 位事件字段限制了单任务直接使用的事件种类，并会合并重复事件。

当任务数量固定且较少、需要携带消息、固定 Tick 开销可以接受时，这套模型依然实用。它的问题不是“不能用”，而是成本会随着任务、定时器和消息数量逐渐暴露。

## 4. LTX V3：以话题就绪队列和闹钟为中心

### 4.1 话题与订阅者

LTX 不先定义任务，而是把“发生了什么”建模为话题。话题持有订阅者链表，同时也是就绪队列中的侵入式节点：

```c
struct ltx_topic {
    uint8_t pending;
    struct ltx_subscriber subscriber_head;
    struct ltx_subscriber *subscriber_tail;
    struct ltx_topic *next;
};

struct ltx_subscriber {
    void (*callback)(void *self);
    struct ltx_subscriber *prev;
    struct ltx_subscriber *next;
};
```

订阅者对象通常嵌入业务结构体，回调再通过 `container_of()` 找回业务对象。这种做法不需要为每次发布动态申请内存，生命周期由应用静态管理。

话题本身不携带事件负载。它更像“某份共享状态已经更新”的通知：生产者先更新数据，再发布话题；订阅者收到通知后读取对应状态。若必须保留每次数据快照，仍需另外设计环形缓冲区或消息队列。

### 4.2 发布与调度路径

发布话题时，LTX 将其 `pending` 置 1；如果话题尚未入队，就追加到就绪队列尾部：

```c
void topic_publish(topic_t *topic)
{
    enter_critical();
    topic->pending = 1;

    if (!topic_is_queued(topic)) {
        ready_tail->next = topic;
        ready_tail = topic;
        request_schedule();
    }
    exit_critical();
}
```

调度器从队首弹出话题，先清除 `pending`，再按订阅顺序调用所有订阅者：

```c
while ((topic = ready_queue_pop()) != NULL) {
    if (!topic->pending) {
        continue;
    }

    topic->pending = 0;
    for_each_subscriber_safe(topic, subscriber) {
        subscriber->callback(subscriber);
    }
}
```

“先清零、再回调”是一个关键细节。订阅者执行期间若再次发布同一话题，它可以重新进入队尾，在本轮订阅者处理结束后再调度一轮。

不同话题按首次入队顺序 FIFO 处理，没有任务优先级。话题发布和队首弹出都是 `O(1)`；一个话题的完整分发仍需遍历其 `S` 个订阅者，因此为 `O(S)`。

和 OSAL 事件位一样，同一话题在尚未调度时被连续发布多次，只保留一次待处理状态。这能限制队列增长，但也意味着话题不是计数信号。

### 4.3 差分闹钟链表

LTX V3 删除了逐 Tick 扫描的软件定时器，改用按到期顺序排列的闹钟链表。每个节点不保存绝对到期时间，而只保存与前一个节点的时间差 `diff_tick`。

假设三个闹钟将在当前时刻之后第 5、12、20 Tick 到期，链表保存的是：

```text
head -> A(+5) -> B(+7) -> C(+8)
绝对时间  5        12        20
```

现在插入一个第 9 Tick 到期的闹钟 D，需要拆分 B 原来的差值：

```text
head -> A(+5) -> D(+4) -> B(+3) -> C(+8)
绝对时间  5         9        12        20
```

插入要寻找位置，最坏为 `O(A)`；已知节点的移除为 `O(1)`，但要把被移除节点的差值补给后继节点。每次 SysTick 只需递减链表首节点，到期后弹出它以及所有同一时刻到期、`diff_tick == 0` 的节点，成本为 `O(1 + K)`，其中 `K` 是本次同时到期的闹钟数量。

这是一种典型的空间换时间位置选择：把每 Tick 的全表扫描，转移到较少发生的闹钟插入操作中。若闹钟新增极其频繁且数量很多，`O(A)` 插入仍可能成为成本；但在常见的少量定时业务中，中断路径会更短、更稳定。

### 4.4 Idle 与 Tickless

差分链表的首节点天然给出了“距离最近一次到期还有多久”，因此适合 Tickless：

```c
next_interval = min(first_alarm->diff_tick, systick_max_interval);
program_systick(next_interval);
```

LTX 可以把调度请求映射到最低优先级 PendSV，将普通主循环留作 Idle：

```c
int main(void)
{
    init_system();

    for (;;) {
        __WFI();
    }
}

void PendSV_Handler(void)
{
    ltx_Sys_scheduler();
}
```

这里的 PendSV 只用于抬升调度执行上下文，并没有像 RTOS 那样保存和切换线程上下文。订阅者仍然运行到完成，而且都运行在 Handler 模式下；耗时回调会推迟主循环和后续同级工作。

低功耗能力也不是打开宏就自动完成：PendSV 和 SysTick 的优先级、`__WFI()` 前后的竞态、时钟源切换、唤醒补偿都必须结合芯片验证。当前 V3.3 源码还明确保留了 SysTick 重编程的补偿取舍：启用补偿曾出现小概率提前触发，不补偿则可能带来少量延后。因此需要精确定时的业务不应只依据设计描述，必须测量目标平台上的实际误差。

### 4.5 App、周期任务与 Script

LTX 核心只有话题和闹钟，上层能力由二者组合：

- 周期任务：闹钟到期后重新登记下一次闹钟，再调用业务回调；
- App：集中管理一组周期任务的初始化、暂停、恢复和销毁；
- Script：用显式步骤号保存流程位置，通过闹钟实现非阻塞延时，通过订阅话题实现非阻塞等待。

Script 的使用方式可以抽象为下面的状态机伪代码：

```c
void startup_script(script_t *script)
{
    switch (script->step) {
    case 0:
        power_on_sensor();
        script_wait_delay(script, 1, 10_ms);
        return;

    case 1:
        start_calibration();
        script_wait_topic(script, 2, 100_ms, &calibration_done);
        return;

    case 2:
        if (script_trigger(script) == TRIGGER_TIMEOUT) {
            report_error();
        } else {
            enable_sampling();
        }
        script_finish(script);
        return;
    }
}
```

这已经接近无栈协程：业务能够写成连续步骤，但局部变量不会像线程栈那样自动跨等待点保存，跨步骤状态仍要放在 `script` 或其私有数据中。这里也为后续比较 Protothreads 提供了直接切入点。

### 4.6 LTX 的设计取舍

LTX 把核心缩小为两件事：话题负责传播状态变化，闹钟负责在未来发布话题。核心路径不依赖动态内存，发布和取就绪工作都是常数时间，差分闹钟又为 Idle 与 Tickless 提供了直接依据。

相应的约束是：

- 没有任务优先级，不同话题主要按 FIFO 调度；
- 话题发布会合并，且本身不保存每次事件的数据；
- 一个慢订阅者会阻塞同一话题的后续订阅者和其他就绪话题；
- 差分闹钟新增为 `O(A)`，时间补偿和溢出边界需要仔细验证；
- 运行在 PendSV 时，全部业务回调都必须遵守异常上下文限制；
- 临界区宏应保存并恢复原中断状态，不能假设调用前中断一定开启。

它更像一个轻量的事件分发底座，而不是小型 RTOS。应用是否可靠，很大程度上取决于共享数据、缓冲队列和回调执行预算如何在上层约束。

## 5. 两种架构的横向对比

### 5.1 机制对照

| 维度 | OSAL | LTX V3 |
|------|------|--------|
| 核心对象 | 任务 | 话题、订阅者、闹钟 |
| 调度单位 | 带事件位的任务 | 就绪话题 |
| 调度顺序 | 静态优先级，高数值优先 | 话题首次入队顺序 FIFO |
| 抢占能力 | 无，任务处理函数运行到完成 | 无，订阅者回调运行到完成 |
| 重复通知 | 同一事件位合并 | 同一待处理话题合并 |
| 数据传递 | 事件位或动态消息 | 默认读取共享状态，负载队列由上层提供 |
| 就绪选择 | 扫描任务链表，`O(T)` | 弹出就绪队首，`O(1)` |
| 定时处理 | 每 Tick 扫描全部定时器，`O(A)` | 递减并弹出表头，`O(1 + K)` |
| 定时新增 | 查找并尾插，`O(A)` | 按到期顺序插入，`O(A)` |
| 核心内存 | 任务、定时器、消息依赖 OSAL 堆 | 对象由调用方提供，无需核心堆 |
| 默认空闲行为 | 主循环忙扫描 | 默认忙调度；可用 PendSV + Idle |
| Tickless | 当前实现未提供 | 内建支持，但需做平台时序验证 |
| 流程表达 | 手写任务状态与事件分支 | 可选 Script 组件表达非阻塞步骤 |

其中 `T` 为任务数，`A` 为活跃定时器或闹钟数，`K` 为同一时刻到期的闹钟数。复杂度只描述相应核心操作，不代表整个系统的端到端延迟。

### 5.2 性能不能只看大 O

LTX 把“找就绪工作”从扫描变成弹出队列，但这不自动等于所有场景都更快。真实延迟至少还受下面几项影响：

```text
事件响应时间
  = 当前不可抢占回调的剩余时间
  + 更早进入队列的回调时间
  + 临界区与中断延迟
  + 本次分发自身的执行时间
```

同样，OSAL 的 `O(T)` 在只有 5 个固定任务时可能非常便宜，而一次日志打印或阻塞式总线访问可能比扫描成本大几个数量级。优化前应先测量任务数、事件频率、最长回调时间和休眠占比。

## 6. 两者共同的工程边界

### 6.1 给每个回调设置执行预算

协作式框架最怕“偶尔很慢”的函数。建议至少统计最大执行时间，并把长工作拆成小步骤：

```c
void protocol_work(protocol_t *ctx)
{
    for (unsigned int i = 0; i < MAX_FRAMES_PER_DISPATCH; ++i) {
        if (!try_process_one_frame(ctx)) {
            break;
        }
    }

    if (has_more_frames(ctx)) {
        request_next_dispatch(ctx);
    }
}
```

这种限额处理既防止一个模块独占调度器，也能让高频输入形成可观察的背压，而不是把延迟隐藏在一个无限处理循环里。

### 6.2 区分状态通知与事件计数

事件位和话题都可能合并重复通知。选择通信方式前，应先确定业务语义：

| 业务语义 | 合适做法 |
|----------|----------|
| “配置已经变化，请读取最新值” | 事件位或话题 |
| “至少有一帧可读” | 话题/事件位 + 环形缓冲区 |
| “每个脉冲都必须计数” | 原子计数器 |
| “每条命令及参数都不能丢” | 有界消息队列 |

通知只负责唤醒，真实数据放入有界缓冲区，是裸机事件框架中很实用的组合。必须同时规定缓冲区满时丢最新、丢最旧还是报告故障。

### 6.3 明确中断与前台的所有权

`volatile` 只能阻止部分编译器优化，不能自动提供原子性、互斥或内存顺序。中断与前台共享数据时，要明确：

- 哪一方写、哪一方读；
- 多字节值在目标架构上能否原子访问；
- 临界区是否保存并恢复进入前的中断状态；
- API 能否从中断调用；
- 回调运行在 Thread 模式还是 Handler 模式。

尤其不能把“关中断，操作，开中断”直接当作可嵌套临界区。如果调用者进入函数前本来就处于关中断状态，无条件开中断会破坏上层约束。稳妥的端口层应保存原状态并按原状态恢复。

### 6.4 正确处理空闲休眠竞态

就绪判空与进入休眠之间不能留下丢失唤醒的窗口。Cortex-M 上常见的基本思路是：

```c
disable_irq();
if (ready_queue_is_empty()) {
    __WFI();
}
enable_irq();
```

关中断时到来的中断会进入 pending 状态并唤醒 `WFI`，开中断后再执行 ISR。实际实现还要结合 `SEVONPEND`、`WFE/WFI` 语义、芯片勘误和时钟恢复流程验证，不能只复制几行模板。

## 7. 如何选择与演进

### 7.1 适合 OSAL 的场景

可以优先考虑 OSAL 式模型的情况包括：

- 业务天然按少量固定任务划分；
- 需要任务优先级，但不需要任务抢占；
- 跨模块通信经常携带独立消息负载；
- 已有可接受的固定 Tick 和内存池预算；
- 团队更习惯“任务 + 事件处理函数”的组织方式。

如果采用它，最值得优先补强的是 Idle Hook、消息队列上限、内存分配失败路径和运行时间统计。

### 7.2 适合 LTX 的场景

可以优先考虑 LTX 式模型的情况包括：

- 系统主要由状态变化和外设完成事件驱动；
- 一个数据更新需要通知多个使用者；
- 希望核心不依赖动态内存；
- 对空闲休眠和 Tickless 有明确需求；
- 愿意用共享状态、环形缓冲区和显式步骤状态机组织业务。

如果采用它，应优先建立回调耗时约束、话题数据所有权规则、队列溢出策略，并在目标芯片上测量 Tickless 的时间误差。

### 7.3 何时该使用 RTOS

当系统出现下面的需求时，与其不断给协作式框架增加补丁，不如认真评估 RTOS：

- 多个流程必须保留各自深层调用栈；
- 第三方库只能用阻塞式 API；
- 需要真正的优先级抢占和确定的唤醒策略；
- 需要成熟的互斥量、信号量、优先级继承和线程诊断；
- 团队能够承担每线程栈空间和上下文切换成本。

裸机框架与 RTOS 不是简单的代码量竞争。关键是应用所需的并发语义，是否已经超出“短回调 + 显式状态”的承载范围。

## 8. 后续加入 Protothreads 的方式

### 8.1 保持相同的比较维度

后续分析 Protothreads 时，不只介绍 `PT_BEGIN`、`PT_WAIT_UNTIL` 等宏，而是继续回答本文的统一问题：

- 续执行位置如何保存，局部变量为什么不能自然跨越等待点；
- 条件等待与时间等待由谁驱动；
- 多个 Protothread 如何被轮询或加入就绪机制；
- `switch/case` 与行号技巧对代码结构有哪些限制；
- 它与 LTX Script 的显式步骤状态机有什么差异；
- 它能否与 OSAL 事件或 LTX 话题组合，而不是替代整个调度底座。

届时可以在对照表中新增“控制流表达”和“跨等待点状态”两行。OSAL 偏向事件分支，LTX Script 偏向显式步骤号，Protothreads 则用宏把续执行状态隐藏到看起来更顺序化的代码背后。

### 8.2 当前结论

OSAL 和 LTX 解决的是同一类裸机组织问题，但抽象中心不同：

```text
OSAL：事件属于任务
      定时器 / 消息 ──> 设置任务事件 ──> 按优先级扫描并执行任务

LTX：订阅者属于话题
      中断 / 闹钟 ──> 话题进入就绪队列 ──> FIFO 弹出并广播
```

OSAL 用较完整的任务运行环境换取直接的业务分区和消息能力；LTX 用更小的事件内核换取常数时间就绪操作、静态对象和更自然的低功耗路径。两者都要求回调短小、等待非阻塞、共享数据有明确所有权。

真正决定架构质量的，不是给 `while (1)` 换了一个名字，而是系统是否把事件语义、时间推进、执行预算、内存上限和空闲竞态都变成了可以说明、测量和验证的规则。
