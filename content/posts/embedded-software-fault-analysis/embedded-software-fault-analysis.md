上一篇 [Cortex-M 异常分析](/articles/cortex-m-fault-analysis/) 介绍了异常栈帧、HardFault 定位和常见软件缺陷。本篇继续深入软件侧：不仅关注最终进入了哪一种 fault，还要追查是哪一次越界写、错误释放或同步操作破坏了系统状态。

文中的同步原理适用于常见 RTOS，接口和部分行为以 RT-Thread 为例。不同 RT-Thread 版本以及其他 RTOS 的配置项、递归互斥量和 Hook 接口可能不同，使用时应以目标工程的头文件和内核源码为准。

---

## 1. 软件缺陷为什么难以定位

软件缺陷不一定在发生时立即触发 CPU fault。例如数组越界可能先覆盖相邻对象，数秒后才因函数指针或返回地址损坏而崩溃；死锁和优先级反转则通常不会进入 HardFault，而是表现为任务长期阻塞或系统无响应。

```text
首次错误操作
    -> 内存、栈或同步状态被破坏
    -> 系统继续运行
    -> 被破坏的数据再次被使用
    -> CPU fault、任务阻塞或看门狗复位
```

因此要区分两个位置：

- **异常暴露点**：栈帧 PC、断言或看门狗记录的位置；
- **缺陷发生点**：第一次越界写、错误释放或错误加锁的位置。

异常现场能帮助确定暴露点，但定位此类问题的目标通常是继续向前找到缺陷发生点。

## 2. 堆内存异常

### 2.1 常见类型

| 类型 | 说明 | 常见后果 |
|------|------|----------|
| 越界写 | 写到分配块的前后边界之外 | 覆盖相邻对象或堆管理信息 |
| 释放后使用 | 内存已归还，但旧指针继续访问 | 数据随机变化、再次分配后互相覆盖 |
| 重复释放 | 同一内存块被释放两次 | 空闲链表损坏或分配器断言 |
| 非法释放 | 释放非堆地址或分配块内部地址 | 堆管理结构损坏 |
| 内存泄漏 | 分配后失去引用，无法释放 | 可用堆逐渐减少，最终分配失败 |

堆管理器通常会在用户数据附近保存块大小、空闲状态或链表指针。一处很小的越界写也可能先破坏这些元数据，直到后续执行 `malloc`/`free` 时才暴露。

### 2.2 哨兵值

哨兵值是在用户区前后放置固定标记，分配、释放或巡检时检查标记是否被改写。

```text
[分配信息][前哨兵][用户数据区][后哨兵]
```

这个方法通常需要包装 `malloc`/`free`、替换项目的分配接口，或者直接扩展内存管理器，并不是“malloc 不能被二次封装”。实现时还应保证返回给用户的地址继续满足对齐要求。

哨兵只能证明“边界曾被覆盖”，不能直接指出是哪条指令写坏了内存。若只在释放时检查，发现时间可能已经远离首次越界；若堆元数据已经严重损坏，也不应继续盲目遍历整个堆。

### 2.3 分配与释放 Hook

RTOS 或 C 运行库提供的 malloc/free Hook 可以记录：

- 分配和释放地址；
- 请求大小；
- 当前线程；
- 返回地址或调用位置；
- 时间戳和剩余堆空间。

Hook 本身只是事件入口，不会自动检测越界、重复释放或泄漏。要完成检测，还需要维护活动分配表、哨兵或统计信息。记录逻辑应避免再次动态分配、阻塞或递归调用被 Hook 的接口，否则诊断代码本身可能引入死锁和重入问题。

RT-Thread 的 Hook 能力受版本和配置影响，应先确认目标工程是否启用了相关 Hook，并以当前版本提供的接口为准。

### 2.4 检测方式对比

| 方法 | 原理 | 适用范围与限制 |
|------|------|----------------|
| 哨兵值 | 在分配块边界写入标记并校验 | 轻量；发现时间取决于检查时机 |
| malloc/free Hook | 记录每次分配和释放 | 便于追踪生命周期；检测能力取决于额外记录逻辑 |
| 定时全量遍历 | 周期性检查活动块和哨兵 | 覆盖面广；发现滞后，且要求堆结构仍可安全遍历 |
| 可疑节点插桩 | 在怀疑的操作前后主动校验 | 定位直接；需要先缩小怀疑范围 |
| 数据观察点 | 监控指定地址的首次写入 | 能捕获写入指令；硬件观察点数量有限 |

### 2.5 排查重点

出现堆异常时，优先保留分配器报错前的记录，不要只检查最终崩溃的 `free`。重点核对同一地址的分配与释放是否配对、块大小是否与拷贝长度一致，以及异常地址附近是否存在被改写的链表指针或哨兵。

## 3. 线程栈异常

### 3.1 栈溢出的来源

RTOS 通常为每个线程分配独立栈。以下情况会使实际使用量超过预留空间：

- 调用层次过深或出现递归；
- 局部数组、结构体或格式化缓冲区过大；
- 函数内部使用变长数组或 `alloca`；
- 中断嵌套增加 MSP 使用量；
- 浮点上下文、库函数或编译选项改变单层调用的栈开销。

在常见 Cortex-M RTOS 移植中，线程使用 PSP，异常处理使用 MSP，但这不是仅凭 RTOS 名称就能确定的规则，应以具体移植层为准。线程栈和异常栈需要分别评估。

### 3.2 检测手段

| 方法 | 原理 | 适用范围与限制 |
|------|------|----------------|
| 栈边界魔术字 | 在线程栈边界放置标记并定期检查 | 实现简单；检查前可能已经覆盖相邻数据 |
| 栈水位填充 | 初始化时填充整段栈，统计未被覆盖区域 | 能保留历史最大使用深度；不能替代边界保护 |
| MPU 保护区域 | 在栈增长方向的边界设置禁止访问区域 | 越界时立即产生 fault；依赖 MPU 和正确的任务切换配置 |
| 编译期分析 | 用 GCC `-fstack-usage` 等生成逐函数栈占用 | 可提前发现大栈帧；仍需结合调用图和运行路径 |

原笔记中的“栈顶魔术字”容易产生方向歧义，称为“栈边界魔术字”更准确。栈水位也不是不能感知瞬时峰值：只要峰值写过填充区域，痕迹通常会保留下来；其局限是只能反映已经运行过的路径，而且真正越界后系统状态可能已经损坏。

### 3.3 MPU 保护的异常表现

MPU 使用的是保护区域，不是 MMU 意义上的内存页。对于向低地址增长的栈，通常在栈下边界设置不可访问区域：

```text
低地址
[MPU 禁止访问区域]
[线程栈可用区域    ]  <- SP 向低地址增长
高地址
```

- Cortex-M3/M4：MPU 访问违规通常触发 MemManage Fault；若该 fault 未使能或无法处理，则可能升级为 HardFault；
- Cortex-M0：没有 MPU；
- Cortex-M0+：可选 MPU，但没有独立 MemManage Fault，违规通常进入 HardFault。

如果 MPU 区域数量不足以同时保护全部线程，RTOS 可以在上下文切换时更新当前线程的保护区域。更新过程必须与任务切换保持一致，否则可能把正常访问误判为越界。

### 3.4 编译期栈分析的边界

GCC `-fstack-usage` 生成的 `.su` 文件描述的是逐函数栈帧大小，不会自动给出某个线程入口到最深调用点的总栈需求。还需要结合调用图、中断嵌套和库函数进行累加。

动态栈分配和递归会降低静态结果的确定性；函数指针和间接调用也可能使调用图不完整。因此编译期分析适合提前筛查，运行期仍应保留栈水位或边界检查。

> 推荐组合：开发阶段使用编译期分析和栈水位统计，高可靠场景再增加 MPU 边界保护。三者解决的问题不同，不能相互替代。

## 4. 死锁

死锁是多个执行单元互相等待资源，导致相关线程永久无法继续运行。它通常不会触发 Cortex-M fault，因此仅依赖 HardFault 日志无法发现。

### 4.1 四个必要条件

以下条件同时成立时才可能形成死锁：

| 条件 | 描述 |
|------|------|
| 互斥 | 资源同一时刻只能由一个线程持有 |
| 请求并保持 | 线程持有已有资源时继续申请其他资源 |
| 不可剥夺 | 资源只能由持有者主动释放 |
| 环路等待 | 多个线程形成循环等待链 |

固定全局加锁顺序可以破坏环路等待。使用超时只能让等待者获得退出机会；超时路径还必须释放已持有资源并回滚状态，否则不能自动消除死锁条件。

### 4.2 常见形式

| 类型 | 场景 | 处理方向 |
|------|------|----------|
| 同线程重复获取 | 同一线程再次获取不支持递归的锁或信号量 | 明确对象是否支持递归，检查调用链 |
| ABBA 死锁 | 线程 A 持有锁 1 等锁 2，线程 B 持有锁 2 等锁 1 | 统一加锁顺序 |
| 异常路径未解锁 | 函数提前返回或错误分支漏掉释放 | 集中清理出口，检查所有返回路径 |
| 持锁等待事件 | 持有互斥量时等待另一个线程完成，而对方也需要该锁 | 等待前释放锁，重新划分临界区 |

当前 RT-Thread 文档中的互斥量支持递归获取，而普通信号量没有所有权，同一线程重复获取耗尽的信号量会阻塞。不同版本或其他 RTOS 的行为可能不同，不能把“`rt_mutex` 由 `rt_sem` 实现”当作其一定会自死锁的依据。

### 4.3 现场分析

分析死锁需要保存线程和 IPC 状态，而不只是 CPU 寄存器：

- 每个线程的运行、就绪和阻塞状态；
- 线程正在等待的对象及超时时间；
- 互斥量当前持有者和递归持有计数；
- 各 IPC 对象的等待队列；
- 最近的加锁、解锁和超时记录。

将“线程 -> 等待对象 -> 对象持有者”连成等待关系图，若形成闭环，就能直接看出死锁链。

## 5. 优先级反转

优先级反转与死锁不是同一个问题。典型场景如下：

```text
低优先级线程 L 持有锁
    -> 高优先级线程 H 请求该锁并阻塞
    -> 中优先级线程 M 持续抢占 L
    -> L 无法及时运行并释放锁，H 的等待时间被拉长
```

优先级继承会临时提升锁持有者 L 的优先级，使其尽快完成临界区并释放锁。RT-Thread 的互斥量提供优先级继承，而普通信号量不提供，因此保护共享资源时不应因为二者都能表示 0/1 就混用。

优先级继承只能缓解这种阻塞，不能修复环路等待、无限长临界区或持锁执行阻塞 I/O。设计时仍应缩短持锁时间，避免在持锁期间等待其他事件，并明确嵌套锁的顺序。

## 6. 信号量、互斥量与其他 IPC

### 6.1 信号量与互斥量

| 维度 | 信号量（Semaphore） | 互斥量（Mutex） |
|------|---------------------|-----------------|
| 所有权 | 通常没有所有权，可由其他上下文释放 | 由持有线程拥有，只能由持有者释放 |
| 数值/状态 | 可以是二值或计数值 | 对外表现为锁定/未锁定，递归实现还会维护持有计数 |
| 优先级继承 | 通常不提供 | RT-Thread 互斥量提供；其他 RTOS 需查对应文档 |
| 递归获取 | 普通信号量不支持 | RT-Thread 互斥量支持；其他实现可能不同 |
| 典型用途 | 资源计数、线程间同步和事件通知 | 保护共享资源和临界区 |

基本选用原则：保护共享资源优先使用互斥量；表示资源数量或发送通知时使用信号量。是否允许在中断中调用，以及应该使用哪个 ISR 安全接口，必须按目标 RTOS 的 API 约束处理。

### 6.2 常见 IPC 误用

- 忽略获取接口的返回值，在超时后仍按“已经获得资源”继续执行；
- 在线程仍等待或使用 IPC 对象时将其删除；
- 在不允许阻塞的中断上下文调用等待接口；
- 持有互斥量时执行无限期等待或阻塞发送；
- 把二值信号量当作事件计数器，导致重复事件合并；
- 消息队列发送长度与创建时的消息大小不一致；
- 错误地假设唤醒线程会立即运行，忽略调度和优先级影响。

这类错误不一定立即崩溃。所有带超时或可能失败的 IPC 调用都应检查返回值，并明确失败后资源状态是否需要回滚。

## 7. 竞态条件与重入

线程之间、线程与中断之间同时访问共享数据时，即使没有使用动态内存，也可能因竞态破坏状态。

```c
shared_count++;
```

这条 C 语句通常包含读取、加一和写回多个步骤。如果线程和中断同时执行，就可能丢失一次更新。`volatile` 只能约束编译器对访问的处理，不能把复合操作变成原子操作，也不能替代锁、临界区或原子指令。

常见问题包括：

- 中断修改链表时，线程同时遍历或删除节点；
- 多个线程复用同一静态缓冲区；
- 不可重入库函数被多个上下文同时调用；
- 检查条件与执行操作之间被其他线程改变状态；
- 回调执行期间对象被另一线程释放。

选择保护方式时应考虑调用上下文：线程之间可以使用互斥量，中断与线程之间通常需要短临界区、原子操作或专用的中断安全接口。不要在中断中等待会阻塞的互斥量。

## 8. RTOS 异常现场保存

### 8.1 CPU fault 现场

发生 HardFault、MemManage Fault、BusFault 或 UsageFault 时，应继续保留上一篇文章介绍的内容：

- MSP、PSP 和 EXC_RETURN；
- 硬件栈帧中的 R0-R3、R12、LR、PC 和 xPSR；
- M3/M4 的 CFSR、HFSR、BFAR 和 MMFAR；
- 与固件完全匹配的 ELF、MAP 和构建标识。

### 8.2 RTOS 上下文

为了判断异常与哪个线程、栈或 IPC 对象有关，还应按目标 RTOS 能力保存：

- 当前线程名称、状态、优先级和错误码；
- 线程栈起始地址、大小、保存的 SP 和栈水位；
- 当前等待的 IPC 对象和剩余超时时间；
- 互斥量持有者、持有计数和等待队列；
- 堆总量、剩余量、最低余量及最近的分配记录；
- 最近若干次线程切换和 IPC 操作记录。

运行中的线程与已经切出的线程，寄存器保存位置可能不同；解析任务栈必须匹配所用 RTOS 版本和 CPU 移植层，不能直接假设所有寄存器都采用同一种布局。

### 8.3 系统卡死现场

死锁、优先级反转和部分竞态不会触发 CPU fault。此时需要由看门狗预警阶段、监控线程或外部调试器主动采集线程列表和 IPC 状态。若直接等待最终复位，只保存 HardFault 寄存器往往得不到有效信息。

现场保存代码应避免使用已经损坏的堆、阻塞式 IPC 或复杂日志路径。保存目标也不能与待保存的 SRAM 区域重叠。

## 9. 定位顺序

面对 RTOS 软件异常，可以按下面的顺序缩小范围：

1. 判断是 CPU fault、线程永久阻塞，还是看门狗复位；
2. 保存 CPU 寄存器、当前线程、任务栈和 IPC 状态；
3. 使用同一次构建的 ELF/MAP 定位异常暴露点；
4. 检查相关地址属于线程栈、堆块、全局对象还是外设区域；
5. 对照分配记录、栈水位和等待关系寻找首次破坏位置；
6. 缩小范围后再加入哨兵、数据观察点或操作轨迹进行复现。

对于堆越界和栈破坏，最终 fault 往往只是结果；对于死锁和优先级反转，则可能根本没有 fault。先判断问题属于哪一种失效模型，再选择现场数据和检测手段，通常比直接在 HardFault 处理函数中增加更多打印更有效。
