
## 总体架构

Cortex-M内核使用**多层总线矩阵（Bus Matrix）**，允许多个主设备（Master）并发访问不同从设备（Slave），无需串行等待。

```
主设备 (Masters)          总线矩阵           从设备 (Slaves)
────────────────         ─────────         ─────────────────
Cortex-M Core            │         │ ──────► Flash (AHB)
  I-Code Bus  ──────────►│  AHB    │
  D-Code Bus  ──────────►│  Bus    │ ──────► SRAM (AHB)
  System Bus  ──────────►│ Matrix  │
DMA Controller ─────────►│         │ ──────► APB Bridge ──► 外设
                         └─────────┘
```

> **注意**：I-Code、D-Code、System Bus 是三条独立总线，有严格的地址映射限制：
>
> | 总线 | 用途 | 地址范围 |
> |------|------|----------|
> | I-Code | 取指（fetch） | `0x00000000~0x1FFFFFFF` |
> | D-Code | 访问代码区数据（Flash常量等） | `0x00000000~0x1FFFFFFF` |
> | System Bus | 访问 SRAM、外设、APB域等 | `0x20000000~0xFFFFFFFF` |
>
> 三者不可互换，编译器/链接器需保证代码和数据放置在正确区域。

### I-Code 与 D-Code 的竞争

两条总线独立，但**共享同一个 Flash 存储体**。竞争的根源是 **Prefetch Buffer miss**，而非总线本身冲突：

- Buffer 命中时：I-Code 走缓冲（0次 Flash），D-Code 走 Flash（1次），真正并行，无竞争
- Buffer miss 时：I-Code 下场抢 Flash（1次）+ D-Code 排队（1次），串行，总延迟变为 2倍 Flash 访问时间

没有 miss 就没有竞争。

| 竞争对 | 频率 | 影响 |
|--------|------|------|
| I-Code vs D-Code | 低（Prefetch 缓解） | 偶发 stall，通常可忽略 |
| I-Code vs DMA 读 Flash | 较高（DMA 持续传输） | 中断延迟变长，需关注 |

### DMA 与 I-Code 的竞争

DMA 竞争比 I-Code vs D-Code 严重得多，原因是 **DMA 是持续性突发传输，不像 D-Code 只是偶发单次读**。

DMA 走 AHB 总线矩阵，读 Flash 时与 I-Code 争同一个 Flash 存储体：

```
I-Code ──────────────►┐
                      ├──► Flash 控制器 ──► Flash 存储体
DMA（读 Flash 时）────►┘

DMA（写 SRAM 时）────► System Bus ──► SRAM   ← 不与 I-Code 竞争
```

**为什么 DMA 竞争更严重：**

1. **突发持续占用**：DMA 启动后连续发起多次读（INCR burst），不主动让出，I-Code 每次 miss 都可能撞上正在传输的 DMA
2. **不受 Prefetch Buffer 保护**：Buffer 命中时 I-Code 不去 Flash，但 DMA 持续传输期间 I-Code 迟早会 miss
3. **带宽持续被占**：AHB 总线矩阵中 CPU 与 DMA 的仲裁优先级是**硬件固定**的（CPU 通常高于 DMA），HAL 中配置的 `DMA_PRIORITY_VERY_HIGH` 只是 DMA 控制器**内部多个通道之间**的优先级，与 AHB 总线矩阵仲裁无关。bus starvation 的根源是 DMA 持续占用带宽，而非优先级反超 CPU

> HAL DMA 优先级 vs AHB 总线矩阵优先级：
> - `hdma.Init.Priority`：DMA1/DMA2 内部，通道与通道之间的仲裁，HAL 可配置
> - AHB 总线矩阵优先级：CPU vs DMA 整体的仲裁，硬件固定，HAL 无法修改
>
> AHB 总线矩阵优先级由芯片设计者在硬件中固化，记录在 Reference Manual 的 Bus Matrix 章节，使用者只能查阅，无法修改。以 STM32F4 为例：
> ```
> 固定优先级（由高到低）：
>   Cortex-M4 I-Code > D-Code > System Bus > DMA1 > DMA2
> ```
> 因此使用者能做的只有**绕开竞争**（让 DMA 不读 Flash、ISR 放 SRAM），而无法改变仲裁规则本身。

**典型影响场景：**

| 场景 | 后果 |
|------|------|
| DMA 从 Flash 搬数据到 SRAM | I-Code 取指被持续延迟 |
| DMA 传输期间触发中断 | 中断延迟 = wait state + DMA burst 剩余时间 |
| 高带宽 DMA（DCMI/I2S/SAI） | Flash 长期被占，CPU 取指持续 stall，整体性能下降 |

**解决方式：**

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| DMA 源改为 SRAM，不读 Flash | 彻底消除与 I-Code 的 Flash 竞争 | 数据预加载到 SRAM |
| ISR 搬到 SRAM（`.ramfunc`） | I-Code 走 System Bus，不争 Flash | 时间敏感中断 |
| 开启 ART 指令 cache | miss 时命中 cache 而非等 Flash | STM32F4+ |
| 限制 DMA burst 长度 | 缩短单次占用时间，给 I-Code 让缝隙 | 实时性要求高 |
| DMA 优先级不设极高 | 避免 CPU bus starvation | 通用原则 |

最根本的解法：**让 DMA 数据源不在 Flash，或让 ISR 不依赖 Flash 取指**，两者择一即可消除竞争。

### 不命中时 I-Code 与 D-Code 的竞争与解决

缓冲 miss 后 I-Code 必须亲自去 Flash 取指，若此时 D-Code 也在读 Flash 常量，两者同时请求，Flash 控制器仲裁（通常 I-Code 优先）。注意不是"打断"，而是**排队**——后到的一方在外面等，前者完成后才能访问：

```
命中时（并行）：
  I-Code → 命中 Buffer，不访问 Flash ──────────────► 取到指令
  D-Code → 直接访问 Flash ────────────────────────► 取到数据
  两者并行，互不影响

不命中时（串行）：
  I-Code → miss，访问 Flash（优先）──► 取到指令
  D-Code → 被阻塞，等 I-Code 完成  ──────────────► 取到数据
  原本并行 → 退化为串行，延迟叠加，共 2 次 Flash 访问
```

**影响本质**：命中时 I-Code 走缓冲、D-Code 走 Flash，两者真正并行；miss 后并行性被破坏，退化为串行，延迟翻倍。

**硬件层面无法完全消除**，但有以下缓解方式：

**1. 把热点 `const` 数据搬到 SRAM**

D-Code 改走 System Bus 读 SRAM，脱离 Flash，miss 时 Flash 全部让给 I-Code：

```c
__attribute__((section(".data"))) const uint16_t sin_table[256] = { ... };
```

**2. 把跳转密集的函数搬到 SRAM（`.ramfunc`）**

I-Code 改走 System Bus，Flash 全部让给 D-Code，竞争方向消除。

**3. 开启 ART Accelerator（STM32F4/H7）**

ART 在 Prefetch Buffer 之上增加了指令 cache 和数据 cache，重复执行的函数跳转 miss 可命中 cache 而无需访问 Flash：

```c
FLASH->ACR |= FLASH_ACR_ICEN     // 指令 cache
           |  FLASH_ACR_DCEN     // 数据 cache
           |  FLASH_ACR_PRFTEN;  // Prefetch
```

**各方案对比：**

| 方案 | 消除 I-Code 竞争 | 消除 D-Code 竞争 | 代价 |
|------|----------------|----------------|------|
| const 数据搬 SRAM | 否 | 是 | 消耗 SRAM |
| 函数搬 SRAM（ramfunc） | 是 | 否 | 消耗 SRAM |
| 两者都搬 SRAM | 是 | 是 | SRAM 消耗较大 |
| 开启 ART Cache | 部分（cache 命中时） | 部分（cache 命中时） | 需 STM32F4+ |

实际项目优先开启 ART，再针对性地把最热的 ISR 或查找表搬 SRAM。



Prefetch Buffer 是 Flash 控制器里的小缓冲，超前于 PC 预读后续指令。CPU 取指时若目标地址已在缓冲中则直接返回（0 wait state），否则需等 Flash。STM32 典型为 **2×64-bit**，可缓存 4 条 32-bit 指令。

```
Buffer：[ 指令A | 指令B | 指令C | 指令D ]
PC 执行 B 时 Buffer 已预读到 D，取 C/D 直接命中
```

**不命中的场景：**

| 场景 | 原因 |
|------|------|
| 跳转指令（B/BL/BX） | PC 突变到缓冲范围外，缓冲全部失效 |
| 中断/异常入口 | 硬件自动跳转 ISR，等同于跳转 |
| 条件分支 taken | 跳转发生时 PC 改变，缓冲失效 |
| 函数返回（BX LR） | 返回地址可能不在缓冲中 |

**命中率规律：**
- 顺序执行、循环体 → 高命中，接近 0 wait state
- 跳转密集、状态机、中断频繁 → 低命中，实际速度接近 Flash 原始速度

C 语言各语句对命中率的影响：

| 语句 | 跳转特征 | 命中率 |
|------|---------|--------|
| 顺序代码 | 无跳转 | 最高 |
| for/while 紧凑循环 | 固定回跳，ART 可缓存 | 较高 |
| if/else | 条件跳转，成立时 miss，不成立时命中 | 中 |
| switch 多分支 | 跳转表（TBB/TBH），地址分散 | 较低 |
| 函数调用（BL/BX） | 跳转+返回，地址不定 | 低 |
| 中断 | 硬件强制跳转，缓冲必然失效 | 最低 |

> 循环体足够小时，整个循环能放进 ART cache，第二圈起回跳命中 cache，不访问 Flash，这是紧凑循环性能好的原因。

这也是嵌入式常见性能优化手法的底层原因：
- **循环展开**：减少回跳次数
- **`inline` 函数**：消除调用跳转
- **减少 switch 分支**：避免跳转表分散跳转
- **热点函数放 SRAM（`.ramfunc`）**：跳转目标在 SRAM，0 wait state，绕过 Flash 竞争



这是**哈佛架构**的体现。若取指和数据访问共用一条总线，两者会互相阻塞：

```
单总线（假设）：取指 → [等待] → 读常量 → [等待] → 取指 ...  串行
双总线（实际）：I-Code 取指 ┐
               D-Code 读常量┘  并行，互不阻塞
```

### I-Code 取指详解

I-Code 取指**不会把代码搬到任何地方**，指令直接在 Flash 原地执行（XIP，Execute In Place）。CPU 内部 PC 指向当前指令地址，I-Code 总线持续从该地址取指，送入流水线：

```
Flash (0x08000000)          CPU 内部流水线
┌─────────────────┐         ┌────────┬────────┬─────────┐
│ 指令1  (32-bit) │──ICode─►│ Fetch  │ Decode │ Execute │
│ 指令2  (32-bit) │         └────────┴────────┴─────────┘
│ 指令3  (32-bit) │           ▲ PC 自动递增，持续取指
└─────────────────┘
```

**每次取多少：**

I-Code 总线是 32-bit 宽，每次取 32-bit。Cortex-M 支持 Thumb-2 指令集：
- 16-bit 指令：一次取回2条
- 32-bit 指令：一次取回1条

Flash 控制器通常有**预取缓冲（Prefetch Buffer）**，一次从 Flash 读 64-bit 或更宽，提前缓存后续指令，CPU 命中缓冲时 0 wait state，无需再访问 Flash。

**为什么不把所有代码都搬到 SRAM 执行：**
- **容量不够**：SRAM 远小于 Flash（如 STM32F407：Flash 1MB vs SRAM 192KB），程序本体放不下，剩余空间还要留给堆栈和变量
- **掉电丢失**：SRAM 是易失性存储，每次上电内容为随机值，代码的源头永远在 Flash，startup 每次都要重新搬运
- **收益有限**：大多数代码（初始化、后台任务）对 wait state 不敏感，没有搬运的必要

因此 `.ramfunc` 只用于**少量时间极敏感的 ISR**，是权衡后的例外优化，不是常规做法。

### 中断函数搬到 SRAM 为何更快

关键在于 **Flash 有 wait state，SRAM 没有**。

Flash 物理读取速度跟不上高频 CPU。以 STM32F4 @168MHz 为例，需要 5 个 wait state：

```
从 Flash 取一条指令：
─┬──┬──┬──┬──┬──┬──►
  W  W  W  W  W  取到     W = CPU 空转等待周期
  ←─ 5 wait states ─►

从 SRAM 取一条指令：
─┬──►
  取到                     0 wait state，直接拿到
```

Prefetch Buffer 在顺序执行时能缓解，但中断发生时 CPU 跳转到新地址，缓冲全部失效，必须重新从 Flash 等待取指。

**实际效果对比（STM32F4 @168MHz，5 wait state，ISR 含 50 条指令）：**

| 执行位置 | 每条指令周期数 | 50条合计 | 时间 |
|----------|--------------|---------|------|
| Flash | 1 + 5 = 6 | 300 周期 | ≈ 1.8 µs |
| SRAM | 1 | 50 周期 | ≈ 0.3 µs |

**使用方式：**

```c
__attribute__((section(".ramfunc"))) void TIM1_IRQHandler(void) {
    // 此函数由链接脚本放入 SRAM，启动时随 .data 段一起搬运
}
```

链接脚本将 `.ramfunc` 的 LMA 放 Flash、VMA 放 SRAM，startup 的 `.data` 搬运时一并复制，之后 CPU 从 SRAM 地址执行。

**D-Code 的使用时机：**

D-Code 是 CPU 在**运行时**通过 load/store 指令访问代码区数据的通路，程序运行期间持续发生，并非只在启动时使用：

| 场景 | 说明 |
|------|------|
| 读取 Flash 中的 `const` 变量、查找表、字符串字面量 | 最常见，贯穿整个运行期 |
| 读取中断向量表（默认在 `0x00000000`） | 每次中断响应都会触发 |
| 函数内 literal pool 取值（编译器生成） | 取立即数常量时由编译器自动插入 |
| 启动时 `.data` 段从 Flash 复制到 SRAM（读侧） | 仅发生一次，写侧走 System Bus |

> 启动时的 `.data` 搬运只是 D-Code 的场景之一。搬运完成后，D-Code 仍会在运行期持续被使用。

**启动时各段的处理方式：**

| 段 | Flash 中有内容？ | 启动时操作 | 涉及总线 |
|----|----------------|-----------|----------|
| `.text` | 是 | 不搬，直接从 Flash 执行 | I-Code（取指） |
| `.rodata` | 是 | 不搬，运行时按需读取 | D-Code |
| `.data` | 是（存初始值） | startup 搬到 SRAM | 读→D-Code，写→System Bus |
| `.bss` | 否，仅记录大小 | startup 将 SRAM 对应区域清零 | System Bus |

```c
// startup 等效逻辑
memcpy(&_sdata, &_sidata, &_edata - &_sdata);  // .data: Flash → SRAM
memset(&_sbss, 0, &_ebss - &_sbss);            // .bss:  SRAM 清零，不读 Flash
```

> `.bss` 在 Flash 中没有内容，只有链接脚本记录的地址和大小，启动时直接清零 SRAM，不涉及任何 Flash 读取。

**各操作的总线归属：**

| 操作 | 总线 |
|------|------|
| 取指令 | I-Code |
| 读 Flash 里的 `const` 数据 | D-Code |
| 启动时从 Flash 复制 `.data` 到 SRAM（读） | D-Code |
| 启动时从 Flash 复制 `.data` 到 SRAM（写） | System Bus |
| 读写 SRAM | System Bus |

**实际影响：**
- `const` 查找表放 Flash → 走 D-Code，不占 System Bus，不与 SRAM 访问竞争
- 中断向量表默认在 `0x00000000` → 走 D-Code 读取，不影响取指流水线
- 若将向量表重映射到 SRAM（`SCB->VTOR = 0x20000000`）→ 改走 System Bus，与其他 SRAM 访问共享带宽

## AHB 仲裁

AHB（Advanced High-performance Bus）是高速总线，连接内核、DMA、SRAM、Flash等。

**仲裁机制：**

- **固定优先级 + 轮询（Round-Robin）**：当多个主设备同时请求同一从设备时，仲裁器介入。
- Cortex-M的总线矩阵为每对主-从路径提供独立通道，**尽量避免冲突**。
- 当冲突不可避免时，按优先级决定，被阻塞的主设备通过 `HREADY = 0` 插入等待。

**DMA优先级与总线饥饿问题：**

DMA 优先级是可软件配置的，**不一定低于 CPU**。以 STM32 为例：
- DMA 通道优先级可设为：低 / 中 / 高 / 极高
- 若多个高优先级 DMA 持续占用总线（如音频流、摄像头 DCMI），CPU 的 System Bus 访问会被持续推迟，造成 **bus starvation**
- 实际项目中需合理规划 DMA 优先级，避免 CPU 被饿死导致实时任务超时

**关键信号：**

| 特性 | 说明 |
|------|------|
| 总线宽度 | 32-bit |
| 流水线 | 地址与数据相位分离，支持流水 |
| 突发传输 | 支持 INCR/WRAP burst |
| 仲裁信号 | `HBUSREQx`（请求）、`HGRANTx`（授权）、`HREADY`（就绪） |

## APB 仲裁

APB（Advanced Peripheral Bus）是低速外设总线，通过 **AHB-to-APB Bridge** 挂接在AHB上。

**APB 本身没有仲裁机制**，原因：

- APB 是单主总线（Bridge 是唯一主设备），不存在多主竞争，从设计上不需要仲裁。
- Bridge 将 AHB 事务顺序转发给 APB 从设备，一次只访问一个。
- 访问期间 Bridge 在 AHB 侧保持 `HREADY = 0`，完成后释放。

**APB 的固有代价：延迟叠加**

没有竞争，但多个外设访问请求在 AHB 侧排队，后来的请求必须等前面事务走完：

```
CPU 连续访问 3 个 APB 外设：
UART → [2周期] → TIM → [2周期] → ADC → [2周期]
串行执行，总耗时叠加
```

访问 APB 外设越频繁，AHB 侧等待越长，这是 APB 单主架构的固有代价，不是竞争问题。

**延迟叠加的根本原因：**

APB Bridge 串行工作 + APB 协议每次最少 2 周期，两层叠加：

```
第一层：APB 协议固定最少 2 周期/次（Setup + Access）
第二层：Bridge 串行转发，前一笔未完成不启动下一笔

CPU 访问 N 个外设：总耗时 = N × 2周期 + 各外设 wait state 之和
```

**如何避免：**

| 方法 | 减少的是 | 适用场景 |
|------|---------|---------|
| 减少不必要的寄存器读写 | APB 事务总数 | 通用，尤其避免 HAL 函数内部多余读操作 |
| DMA 代替 CPU 轮询 | CPU 访问 APB 的频率 | UART/SPI/ADC 等数据流外设 |
| 批量写寄存器（一次写多位） | 多次事务合为一次 | 初始化、GPIO 配置 |
| 配置移到初始化阶段 | 运行期 APB 访问次数 | 固定参数场景 |
| 选挂 AHB 的外设（如 STM32H7 GPIO 在 AHB4） | Bridge 串行开销 | 选型阶段 |

```c
// 差：3次 APB 写
GPIOA->MODER |= (1 << 0);
GPIOA->MODER |= (1 << 2);
GPIOA->MODER |= (1 << 4);

// 好：1次 APB 写
GPIOA->MODER |= (1<<0) | (1<<2) | (1<<4);
```

**HAL 库会加重延迟叠加**

HAL 的封装使每次函数调用内部隐含多次 APB 读-改-写，表面上一行代码，实际是多次 APB 访问：

```c
// 连续调用 3 个 HAL 使能，每个内部各有 1次读 + 1次写：
__HAL_UART_ENABLE_IT(&huart, UART_IT_RXNE);  // APB 读 + 写
__HAL_UART_ENABLE_IT(&huart, UART_IT_TXE);   // APB 读 + 写
__HAL_UART_ENABLE_IT(&huart, UART_IT_IDLE);  // APB 读 + 写
// 共 6 次 APB 访问

// 合并为直接寄存器操作：
USART1->CR1 |= USART_CR1_RXNEIE | USART_CR1_TXEIE | USART_CR1_IDLEIE;
// 共 2 次 APB 访问（1读 + 1写）
```

HAL 的价值在于可移植性和可读性，不在于性能。初始化阶段和运行期性能敏感路径上，直接操作寄存器并合并位操作是更好的选择。

**APB访问时序（标准两周期，无wait state）：**

```
        Cycle 1 (Setup)   Cycle 2 (Access)
CLK     ───┐ ┌───┐ ┌───┐ ┌───
           └─┘   └─┘   └─┘
PSEL    ───────────────────────  (Bridge拉高，持续整个事务)
PENABLE ─────────────────        (第2周期才拉高)
PREADY  ──────────────────       (PENABLE为高时由从设备给出；若为低则插入等待周期)
PWRITE  ───────────────────      (读/写方向，Setup阶段建立)
```

- **Setup phase**（第1周期）：PSEL=1，PENABLE=0，地址/控制信号建立
- **Access phase**（第2周期起）：PENABLE=1，等待 PREADY=1 后事务结束
- 若从设备未就绪，拉低 PREADY 可插入任意数量等待周期

---

## 两者协同

1. CPU/DMA发起AHB请求 → 总线矩阵仲裁
2. 若目标在APB域 → AHB矩阵路由到APB Bridge
3. Bridge独占APB，完成事务后归还AHB控制权
4. AHB期间CPU侧 `HREADY=0`，整个通路串行化

**外设挂在APB的原因：**

定时器、ADC、UART等外设挂APB，核心原因是**访问频率低、带宽需求小**，用低速总线节省功耗和布线资源。并非因为"时钟敏感"——事实上定时器的计数时钟往往来自APB时钟的倍频（如STM32中 TIMx_CLK 在 APB 预分频器不为1时等于 PCLK×2），与APB访问速度无关。

## 日常开发常见异常

### AHB 相关

**1. DMA 与 CPU 访问同一 SRAM 数据不一致**
- DMA 写入 buffer 后，CPU 读到的仍是旧数据（或反过来）
- 原因：CPU 有 write buffer / 流水线，写操作未及时刷新到 SRAM
- 解决：传输前后加内存屏障 `__DSB()` / `__DMB()`，或将 DMA buffer 标记为 non-cacheable

**2. 非对齐访问 HardFault**
- AHB 要求自然对齐（32-bit 访问地址需4字节对齐）
- 常见场景：将 `uint8_t*` 强转为 `uint32_t*`，或 `#pragma pack` 结构体跨总线传输
- Cortex-M0/M0+ 硬件不支持非对齐；M3/M4 默认支持，但可通过 `CCR.UNALIGN_TRP` 禁用

**3. Flash 取指与 DMA 读 Flash 竞争**
- CPU I-Code 和 DMA 同时访问 Flash，仲裁导致 CPU 取指 stall
- 后果：中断延迟变长，实时性变差
- 解决：将时间敏感的中断处理函数放 SRAM（`__attribute__((section(".ramfunc")))`）

**4. AHB Bus Fault / Lockup**
- 访问未映射地址或未使能的外设触发 Bus Fault
- 若 Bus Fault handler 本身也出错 → Lockup，芯片停机只能复位
- 定位：读 `BFAR`（Bus Fault Address Register）和 `CFSR` 获取出错地址和原因

---

### APB 相关

**1. 外设时钟未使能就访问寄存器**
- APB Bridge 转发后，外设无时钟响应 → PREADY 永不拉高 → CPU 卡死或 Bus Fault
- 最常见低级 bug：忘记 `RCC_APB1ENR |= ...`

**2. 读-改-写竞争（RMW hazard）**
- APB 访问最少2周期，中间若有中断打断并修改同一寄存器，写回时会覆盖中断的修改
- 常见于 GPIO ODR、定时器 CR1 等多 bit 共享寄存器
- 解决：用 BSRR（GPIO 原子置位/清零）、原子位操作寄存器，或关中断保护临界区

**3. APB 分频后定时器频率计算错误**
- STM32 APB 预分频器不为1时，`TIMx_CLK = PCLK × 2`，若按 PCLK 计算频率，定时周期差一倍
- 症状：定时器周期是预期的一半或两倍

**4. 外设复位后未重新初始化**
- 通过 RCC APB reset 复位外设后寄存器恢复默认值，若不重新初始化直接使用会行为异常
- 常见于 fault 恢复流程或动态重配置场景

---

### 跨域问题（AHB ↔ APB）

**DMA 搬运 APB 外设数据的节拍对齐**
- DMA 从 APB 外设（如 ADC DR、UART DR）读数据时，APB 访问需2+周期
- DMA 请求信号（REQ）和外设数据就绪时序若对不上，会读到旧值或触发溢出
- 解决：确认外设的 DMA 请求在数据寄存器稳定后才发出（查手册确认 DMA request 触发时序）
